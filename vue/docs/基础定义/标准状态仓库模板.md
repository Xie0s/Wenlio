# 标准状态仓库模板

## 背景与目标

适用范围：新增业务 Store（如用户、配置、页面状态、缓存数据）。

目标是统一状态管理写法，避免各模块实现风格分散，降低后续维护成本。

## 规范项

### 1) 标准模板结构

```ts
/**
 * useXxxStore.ts - 业务状态管理模板
 * 职责：管理 xxx 业务状态、派生状态与业务动作
 * 对外暴露：useXxxStore
 */
import { defineStore } from 'pinia'

interface XxxState {
  list: string[]
  loading: boolean
  lastUpdatedAt: number | null
}

export const useXxxStore = defineStore('xxx', {
  // 1) 共享状态
  state: (): XxxState => ({
    list: [],
    loading: false,
    lastUpdatedAt: null,
  }),

  // 2) 派生数据
  getters: {
    hasData: (state) => state.list.length > 0,
    count: (state) => state.list.length,
  },

  // 3) 业务动作
  actions: {
    setList(payload: string[]) {
      this.list = payload
      this.lastUpdatedAt = Date.now()
    },
    setLoading(value: boolean) {
      this.loading = value
    },
    reset() {
      this.list = []
      this.loading = false
      this.lastUpdatedAt = null
    },
  },

  // 4) 持久化（pinia-plugin-persistedstate@4.x）
  persist: {
    key: 'xxx-store',
    pick: ['list'],
  },
})
```

### 2) 字段持久化建议

- `persist.pick` 仅持久化必要字段。
- 不持久化 `loading`、临时 UI 状态等瞬时数据。

### 3) 目录与职责边界

- Store 放在 `src/stores` 目录。
- 组件层只负责 UI 与交互。

## 示例（落地检查）

- 是否具备 `state/getters/actions/persist` 四段结构。
- 是否仅持久化业务必要字段。
- 是否避免把业务逻辑放入组件层。

## 维护建议

- 新增业务 Store 时优先从该模板复制并按业务裁剪。
- 若需扩展持久化字段，先评估是否会引入旧数据兼容问题。
