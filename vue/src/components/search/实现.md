### 并发性能：文档集多时的效果

搜索一次命中 10 篇文档，这 10 篇来自 **N 个不同版本**（最坏情况 N=10）：

**旧实现（串行）**：
```
查version1 → 查theme1 → 查version2 → 查theme2 → ... → 查version10 → 查theme10
耗时 ≈ 10 × 2 × RTT（假设每次 MongoDB 往返 2ms → 约 40ms 额外开销）
```

**新实现（并发）**：
```
goroutine1: 查version1 → 查theme1  ┐
goroutine2: 查version2 → 查theme2  ├─ 全部并行，WaitGroup 等待最慢的
...                                 │
goroutine10: 查version10 → 查theme10 ┘
耗时 ≈ 1 × max(单次 RTT) ≈ 仅 2–4ms 额外开销
```

### 实际限制与建议

**本方案天花板：MongoDB `$text` 索引**
- 中文分词依赖 MongoDB 自带的 text tokenizer，**不支持中文分词**（把每个字当独立 token），因此搜"文档管理"实际是搜两个独立字，召回率有限
- 对英文/拼音效果更好

**文档集极多时（>10万篇）**：
- `CountDocuments` 在 `$text` 查询上需要全扫描匹配集，响应时间会上升
- 建议开启 MongoDB Atlas Search（基于 Lucene，支持中文分词）或接入 Elasticsearch，后端的 [SearchService](file:///e:/%E6%96%87%E6%A1%A3%E7%AE%A1%E7%90%86/doc-mather-1003/backend/internal/service/search_service.go:41:0-45:1) 架构已经隔离得足够干净，届时只需替换 [pageRepo.TextSearch](file:///e:/%E6%96%87%E6%A1%A3%E7%AE%A1%E7%90%86/doc-mather-1003/backend/internal/repository/page_repo.go:144:0-171:1) 内部实现即可，上层 handler/前端无需改动

**当前方案对中小规模（<5万篇）是够用的**，并发改造已将元信息获取的额外延迟压到最低，整体搜索响应通常在 **50-200ms** 内。

## 性能分析与优化总结

### 1. 元数据查询：2N 次 → 固定 2 次 DB 往返 ⚡ 最大收益

| | 改前 | 改后 |
|---|---|---|
| 方法 | [fetchVersionMetaConcurrent](file:///e:/%E6%96%87%E6%A1%A3%E7%AE%A1%E7%90%86/doc-mather-1003/backend/internal/service/search_service.go:141:0-177:1)：每个 unique version 一个 goroutine，各自查 version + theme | [fetchVersionMeta](file:///e:/%E6%96%87%E6%A1%A3%E7%AE%A1%E7%90%86/doc-mather-1003/backend/internal/service/search_service.go:140:0-181:1)：1次 `$in` 查所有 version，1次 `$in` 查所有 theme |
| 10条结果跨3版本 | 6次并行 DB 查询 | **2次** DB 查询 |
| 代码复杂度 | goroutine + WaitGroup + Mutex | 纯顺序代码 |

**goroutine 在这里反而是负担**：多个 goroutine 各自占用连接池连接，调度开销 > 并发收益。2次批量查询更简洁且更快。

### 架构层面的长期优化建议

以下改动需要修改 Page 实体结构，当前未实现，但收益最大：

- **预存 `excerpt` 字段**：在发布文档时，将内容 strip markdown 后存前300字。搜索时直接读取，无需每次请求都处理全文。这能让 `content` 字段从 `$project` 中彻底移除，`$facet` 数据传输量下降 90%+。
- **`title` 前缀索引**：`{tenant_id: 1, title: 1}` 复合索引能让标题正则匹配走索引扫描（仅对 `^pattern` 前缀正则有效）；正文 regex 无法避免全扫描，但 `$match` 先过滤了 `tenant_id + status` 已大幅缩小扫描集。
